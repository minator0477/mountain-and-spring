# 地理空間ファイルフォーマット比較: GeoPackage・FlatGeobuf・GeoParquet

## 概要

| 項目 | GeoPackage | FlatGeobuf | GeoParquet |
|------|-----------|------------|------------|
| 拡張子 | `.gpkg` | `.fgb` | `.parquet` |
| ベース形式 | SQLite DB | FlatBuffers バイナリ | Apache Parquet |
| 標準化 | OGC 標準 | 非公式（事実上の標準） | OGC 標準（GeoParquet 1.0） |
| 読み書き | ランダムアクセス | シーケンシャル中心 | カラム型・列指向 |
| 空間インデックス | 内蔵（R-tree） | ヒルベルト曲線ソート | 外部ツール依存 |
| 複数レイヤー | 対応 | 非対応（1ファイル1レイヤー） | 非対応（1ファイル1レイヤー） |
| 主な用途 | デスクトップGIS・汎用 | ストリーミング・Web配信 | 大規模データ分析 |

---

## 各フォーマット詳細

### GeoPackage (`.gpkg`)

**概要**
OGC（Open Geospatial Consortium）が策定したオープン標準フォーマット。
内部は SQLite データベースであり、ベクタ・ラスタ・属性テーブルを1ファイルに格納できる。

**特徴**
- QGIS・ArcGIS・PostGIS など主要GISツールで広くサポート
- 複数レイヤー（ポイント・ライン・ポリゴン混在）を1ファイルにまとめられる
- 内蔵の R-tree 空間インデックスにより、バウンディングボックスフィルタが高速
- トランザクション・更新・削除が可能（編集可能なDB）
- ファイルサイズは中程度

**向いているケース**
- デスクトップGIS での作業・データ配布
- 複数レイヤーをまとめて渡したい場合
- 非エンジニアのGISユーザーとのデータ受け渡し
- Shapefile の代替（既存ワークフローの移行先）

**注意点**
- SQLite の性質上、並列書き込みには不向き
- ファイルが大きくなると操作が重くなる場合がある

---

### FlatGeobuf (`.fgb`)

**概要**
FlatBuffers（Googleのシリアライズライブラリ）をベースにした高速バイナリフォーマット。
Shapefile の後継として設計されており、ストリーミング配信・Web配信を念頭に置いている。

**特徴**
- ヒルベルト曲線によるソートで空間的に近いデータが連続配置され、バウンディングボックス検索が高速
- HTTP Range Request を使ったサーバサイドフィルタが可能（クラウドネイティブ）
- ファイルサイズが小さく、読み込みオーバーヘッドが低い
- GDAL/OGR・Leaflet・MapboxGL JS などで対応済み
- 仕様がシンプルで実装しやすい

**向いているケース**
- Webブラウザへのストリーミング配信
- CDN・S3 などのオブジェクトストレージからの直接配信
- 大量のフィーチャを逐次処理するパイプライン
- ファイルサイズを抑えたい場合

**注意点**
- 1ファイル1レイヤーのみ
- ランダム更新・編集には不向き（基本的に書き捨て）
- SQLite のような高度なクエリ機能はない

---

### GeoParquet (`.parquet`)

**概要**
Apache Parquet（列指向フォーマット）に地理情報（WKBジオメトリ・CRS等）のメタデータを付加した形式。
OGC GeoParquet 仕様（v1.0 以降）として標準化されている。

**特徴**
- 列指向のため、特定の属性列だけを読み込む分析クエリが高速
- 優れた圧縮率（Snappy・ZSTD 等）でストレージコストを削減
- DuckDB・Apache Spark・pandas/GeoPandas・BigQuery など分析基盤で直接利用可能
- Parquet の行グループ統計（min/max）を使った列・空間フィルタプッシュダウン
- クラウドオブジェクトストレージ（S3・GCS）との親和性が高い

**向いているケース**
- 大規模データの分析・集計処理（数百万〜数億フィーチャ）
- DuckDB・Spark・BigQuery など分析エンジンとの連携
- データレイクへの地理データ格納
- 属性値の集計・フィルタが多い分析ワークフロー

**注意点**
- 従来のデスクトップGIS（QGISなど）での対応はまだ発展途上
- 空間インデックスは仕様に含まれるが、ツール側の実装に差がある
- ジオメトリ操作よりも属性分析に強みがある

## 選択フローチャート

```
用途は何か？
├── デスクトップGIS での編集・配布
│   └── → GeoPackage
│
├── Webブラウザへのストリーミング配信 / CDN配信
│   └── → FlatGeobuf
│
├── 大規模データ分析（DuckDB / Spark / BigQuery）
│   └── → GeoParquet
│
└── どれか迷う場合
    ├── 複数レイヤーが必要  → GeoPackage
    ├── 高速ストリーミングが必要 → FlatGeobuf
    └── 列単位の集計が必要 → GeoParquet
```

## パフォーマンス特性の目安

| 操作 | GeoPackage | FlatGeobuf | GeoParquet |
|------|-----------|------------|------------|
| バウンディングボックスフィルタ | ◎（R-tree） | ◎（ヒルベルト） | ○（行グループ統計） |
| 全件読み込み | ○ | ◎ | ○ |
| 属性フィルタ・集計 | ○（SQL） | △ | ◎（列指向） |
| ストリーミング（HTTP Range） | △ | ◎ | ○ |
| 編集・更新 | ◎（DB） | △ | △ |
| 圧縮効率 | ○ | ○ | ◎ |

---

## まとめ

- **GeoPackage** → 既存GISワークフロー・Shapefile 代替・複数レイヤー管理
- **FlatGeobuf** → Web配信・ストリーミング・軽量バイナリ転送
- **GeoParquet** → 大規模データ分析・クラウドデータレイク・列指向クエリ

3つを排他的に選ぶのではなく、**パイプラインの段階に応じて使い分ける**のが実践的。
例: 分析処理は GeoParquet → 結果の配布は GeoPackage → Web表示は FlatGeobuf に変換。
